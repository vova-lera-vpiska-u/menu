# Asset Handling

> Status: **active guidance**. This documents how images and other binary assets
> should be authored, compressed, stored, and referenced in this app. Today most
> assets are bundled with the app; the medium-term direction is to move large
> images to **Supabase Storage** (S3-compatible) and keep only tiny UI assets in
> the bundle.

## Guiding principle

The bundle ships on every visit and is precached by the service worker — so every
kilobyte in it is paid for up front, by every user, on every fresh install. Keep
the bundle small and let large, swappable content live behind a URL.

Two levers, applied in this order:

1. **Compress** — the cheapest win, no infra. A 700 KB PNG is almost always a
   100 KB WebP with no visible loss.
2. **Externalise** — move large assets out of the bundle into Supabase Storage so
   they can change without a redeploy and don't bloat the precache.

## Format decision table

| Asset kind | Format | Why |
|---|---|---|
| Photos, illustrations, complex raster art | **WebP** | 25–35% smaller than JPG, supports transparency, universally supported in our target browsers. Default for all bundled/uploaded raster images. |
| Photos where every byte counts and you have AVIF tooling | **AVIF** | ~50% smaller than JPG, but slower to encode and not worth the complexity unless an image is genuinely large. Use selectively. |
| Icons, logos, line art, anything geometric | **SVG** | Vector, infinitely scalable, tiny, and stylable with CSS. Never rasterise an icon. |
| Images needing transparency + sharp edges (UI sprites) | **WebP** (lossless) or **PNG** | Prefer lossless WebP; fall back to PNG only if a tool can't produce WebP. |
| Source/master files | **PNG / original** | Keep an uncompressed master outside the repo (or in design tooling); commit only the optimised output. |

**Avoid:** shipping `.jpg`/`.jpeg` and `.png` for content images when WebP is an
option. Avoid GIF entirely — use a short muted `<video>`/WebP-animation if motion
is needed.

## Compression targets

Rules of thumb for the output you commit or upload. Treat these as ceilings, not
goals — smaller is better as long as it looks right at the size it renders.

| Use | Max dimension | Target weight |
|---|---|---|
| Full-bleed / hero raster | ≤ 1600 px on long edge | ≤ 150 KB |
| Card / element image (e.g. the menu element tiles) | ≤ 800 px | ≤ 80 KB |
| Thumbnail | ≤ 320 px | ≤ 25 KB |
| Inline SVG icon | n/a | ≤ 4 KB |

Always export at the **actual rendered resolution** (×2 for retina at most). Don't
ship a 3000 px image into a 300 px slot.

### Tooling

- **WebP from PNG/JPG:** `cwebp -q 80 in.png -o out.webp` (quality 75–85 is the
  sweet spot for photos; use `-lossless` for flat UI art).
- **Batch:** `for f in *.png; do cwebp -q 80 "$f" -o "${f%.png}.webp"; done`
- **AVIF:** `avifenc --min 20 --max 30 in.png out.avif`
- **SVG:** run through `svgo` before committing.

## Where assets live

### Bundled — `src/shared/assets/`

For **small, static, ship-with-the-app** assets only:

- UI icons / SVGs (e.g. `DropdownArrow.svg`, `CheckArrow.svg`)
- Tiny PNGs that are part of the chrome
- PWA icons (generated by `@vite-pwa/assets-generator` from `public/icon.svg`)

Imported directly so Vite fingerprints and precaches them:

```ts
import DropdownArrow from '@shared/assets/img/DropdownArrow.svg'
```

Keep the total here small — it is all precached by the service worker.

### Externalised — Supabase Storage (S3-compatible)

For **large content images** (the element/dish artwork, future recipe photos).
Supabase Storage is the chosen backend because the app already depends on Supabase
— no new account, no new secrets, and it exposes a standard S3 API if we ever need
it.

- Bucket: `assets` (public read).
- Reference by public URL built from the existing `VITE_SUPABASE_URL` env var —
  do **not** hardcode the project domain.
- These are **not** in the precache; see the PWA section below for offline.

Free-tier limits to stay under: **1 GB stored, 5 GB egress / month.** With
service-worker caching, repeat visits don't re-download, so egress is dominated by
first-time visitors. Compression keeps both numbers low.

> Alternative backends, if traffic or asset size ever outgrows the free tier:
> **Cloudflare R2** (10 GB free, $0 egress forever) or **Backblaze B2**. Both are
> S3-compatible, so the URL-based reference pattern below doesn't change.

## Referencing externalised assets

Build URLs through a single helper so the storage origin lives in exactly one
place (and a future backend swap is a one-line change). Per the project's effector
and naming rules, no `as`, no `!`, no one-letter names.

```ts
// src/shared/lib/asset-url.ts
const STORAGE_BASE = `${import.meta.env.VITE_SUPABASE_URL}/storage/v1/object/public/assets`

export const getAssetUrl = (fileName: string): string => `${STORAGE_BASE}/${fileName}`
```

```tsx
// usage
import { getAssetUrl } from '@shared/lib/asset-url'

<Element imageURL={getAssetUrl('Fire.webp')} to={FIRE_PATH}>
```

Always set `loading="lazy"` and explicit `width`/`height` on `<img>` (or the
equivalent on background-image containers) to avoid layout shift while remote
images load.

## PWA / offline implications

Externalised assets leave the precache, so they must be cached at runtime or the
app breaks offline after the first load. Add a Workbox `runtimeCaching` rule in
[vite.config.ts](../vite.config.ts) for the storage origin:

```ts
workbox: {
    globPatterns: ['**/*.{js,css,html,svg,png,ico,woff,woff2,ttf,otf}'],
    runtimeCaching: [
        {
            urlPattern: ({ url }) => url.pathname.includes('/storage/v1/object/public/assets/'),
            handler: 'CacheFirst',
            options: {
                cacheName: 'remote-assets',
                expiration: { maxEntries: 60, maxAgeSeconds: 60 * 60 * 24 * 30 },
                cacheableResponse: { statuses: [0, 200] },
            },
        },
    ],
}
```

`CacheFirst` is correct because asset URLs are content-addressed by name — a
changed image gets a new filename, so stale cache is never a problem. If you ever
overwrite a file in place, switch that entry to `StaleWhileRevalidate` or version
the filename (`Fire.v2.webp`).

## Workflow: adding a new content image

1. Export the master at the right resolution (see compression targets).
2. Convert to WebP: `cwebp -q 80 master.png -o name.webp`.
3. Verify the output weight is under the target for its use.
4. Upload to the `assets` bucket in Supabase (dashboard drag-drop, or a script).
5. Reference it with `getAssetUrl('name.webp')` — never a hardcoded URL.
6. If it's a brand-new origin or pattern, confirm the runtime-caching rule covers
   it.

## Workflow: adding a small UI asset

1. Prefer SVG; run it through `svgo`.
2. Drop it in `src/shared/assets/img/`.
3. Import it directly. Keep it small — it's precached.
